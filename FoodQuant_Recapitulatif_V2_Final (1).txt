1. PRÉSENTATION DU PROJET
================================================================================

FoodQuant est une application web de nutrition personnalisée qui génère des 
menus quotidiens adaptés au profil de chaque utilisateur (âge, poids, taille, 
sexe, objectif). L'application calcule les besoins caloriques via la formule 
de Mifflin-St Jeor et propose 4 repas par jour (petit-déjeuner, déjeuner, 
collation, dîner) grâce à l'API Spoonacular.


================================================================================
2. TECHNOLOGIES UTILISÉES
================================================================================

Frontend :
  - React (Vite) avec JSX
  - Tailwind CSS v4 pour le style
  - Recharts pour les graphiques du dashboard
  - LocalStorage pour la persistance côté client (user, recipes, JWT)

Backend :
  - PHP 8.2 (serveur intégré via Docker)
  - PDO MySQL pour la base de données
  - API Spoonacular (recettes et nutrition)
  - API Anthropic Claude Haiku (traduction des recettes)
  - Système de cache fichier (recettes + traductions)
  - Authentification JWT (JSON Web Tokens)

Base de données :
  - MySQL (local via Laragon + en ligne via Railway)

Déploiement :
  - Frontend : Vercel (https://foodquant.vercel.app)
  - Backend : Railway (https://foodquant-production.up.railway.app)
  - Base de données : MySQL sur Railway
  - Code source : GitHub (https://github.com/valerie-kergonan/foodquant)

4. BASE DE DONNÉES (MySQL)
================================================================================

Table "utilisateurs" :
  - idutilisateurs (INT, AUTO_INCREMENT, PRIMARY KEY)
  - prenom (VARCHAR 100)
  - email (VARCHAR 255, UNIQUE)
  - mot_de_passe (VARCHAR 255, hashé avec password_hash)
  - age (INT)
  - weight (FLOAT)
  - height (FLOAT)
  - gender (VARCHAR 10)
  - goal (VARCHAR 20) : perte / maintien / muscle
  - diet (VARCHAR 30) : classique / végétarien / végan / sans gluten
  - besoin_calorique (INT)
  - date_inscription (DATETIME)

Table "historique_repas" :
  - id (INT, AUTO_INCREMENT, PRIMARY KEY)
  - utilisateur_id (INT, FOREIGN KEY → utilisateurs)
  - date_repas (DATE)
  - recettes_json (MEDIUMTEXT) : les 4 recettes du jour en JSON
  - total_calories (INT)
  - total_proteines (INT)
  - created_at (DATETIME)
  - UNIQUE KEY (utilisateur_id, date_repas) : un seul menu par jour

6. FONCTIONNALITÉS IMPLÉMENTÉES
================================================================================

6.1 - Authentification & Sécurité
  - Inscription avec email + mot de passe (hashé avec password_hash)
  - Connexion avec vérification (password_verify)
  - Tokens JWT générés à la connexion/inscription (durée 7 jours)
  - JWT envoyé dans le header Authorization: Bearer <token>
  - Validation des entrées côté serveur (email, âge, poids, taille)
  - Identifiants BDD via variables d'environnement (plus de mots de passe en dur)
  - Déconnexion (suppression JWT + localStorage)

6.2 - Profil utilisateur
  - Formulaire : prénom, âge, poids, taille, sexe, objectif, régime
  - Calcul automatique des besoins caloriques (Mifflin-St Jeor)
  - Sélecteur de régime alimentaire avec boutons visuels + emojis
  - Sauvegarde en base de données (protégée par JWT)

6.3 - Génération de menus
  - 4 repas/jour : petit-déjeuner (25%), déjeuner (35%), collation (10%), dîner (30%)
  - Appel API Spoonacular avec type de repas et calories cibles
  - Cache fichier côté serveur (1h) avec mutualisation par tranche de 50 kcal
  - Verrouillage quota (erreur 402) pendant 1h
  - Gestion rate limit (erreur 429) avec fallback immédiat
  - Cache expiré utilisé en secours avant le fallback
  - 20 recettes de secours en français (par type de repas)
  - Bouton "Changer" pour remplacer un repas individuellement

6.4 - Traduction des recettes (V2)
  - Traduction automatique des titres et ingrédients via API Claude Haiku
  - Cache permanent par recette (jamais retraduit)
  - Fallback sur le titre anglais si la traduction échoue
  - Coût quasi nul (~0.001$ par recette)

6.5 - Affichage Ma Journée (V2)
  - Bilan nutritionnel compact : calories, protéines, lipides, glucides, fibres
  - Barres de progression colorées avec détection de dépassement (rouge)
  - Objectifs calculés automatiquement (30% lipides, 50% glucides, 30g fibres)
  - Détail nutritionnel par repas (lipides, glucides, fibres) en dépliant
  - Titres des recettes en français
  - Anneaux de progression par repas (calories + protéines)
  - Animations d'entrée en cascade (fadeSlideUp)
  - Emojis par type de repas (🌅 🍎 ☀️ 🌙)
  - Bouton "Recette" avec lien vers la source
  - Message guide si aucun menu n'est généré

6.6 - Liste de courses (V2)
  - Modale accessible via le bouton 🛒 dans le header
  - Extraction automatique des ingrédients des 4 recettes
  - Priorité aux ingrédients traduits (ingredients_fr)
  - Dédoublonnage automatique des ingrédients identiques
  - Cases à cocher (vert + barré quand coché)
  - Indication du repas d'origine par ingrédient
  - Bouton "Copier la liste" pour coller dans une app de notes
  - Bouton reset pour tout décocher
  - Compteur d'ingrédients + cochés

6.7 - Tableau de bord + Badges (V2)
  - Stats rapides : jours consécutifs (streak), total jours suivis, moyenne kcal
  - Graphique barres : calories par jour (Recharts)
  - Graphique barres : protéines par jour
  - Sélecteur de période : 7j / 14j / 30j
  - Historique détaillé : liste des jours avec calories et protéines
  - 8 badges de gamification :
    🏆 Premier pas (1er menu sauvegardé)
    🌱 Bonne habitude (3 jours de suivi)
    📅 Régulier (10 jours de suivi)
    🔥 En feu ! (7 jours consécutifs)
    💎 Diamant (14 jours consécutifs)
    🌟 Un mois ! (30 jours total)
    👑 Roi du suivi (30 jours consécutifs)
    💪 Bien calibré (moyenne calories dans l'objectif ±15%)
  - Barre de progression des badges (X/8)
  - Badges grisés quand non débloqués

6.8 - Message de bienvenue personnalisé (V2)
  - Après inscription : "🎉 Bienvenue [prénom] ! Commence par remplir ton profil."
  - Après connexion : "👋 Content de te revoir [prénom] !"
  - Toast animé en haut de l'écran (4 secondes)

6.9 - Footer & Conformité RGPD (V2)
  - Footer en bas de la page d'accueil
  - Politique de confidentialité (RGPD) :
    → Données collectées (compte, profil, utilisation)
    → Finalité du traitement
    → Base légale (consentement article 6.1.a)
    → Durée de conservation
    → Sous-traitants (Spoonacular, Anthropic, Railway)
    → Sécurité (bcrypt, JWT, HTTPS)
    → Droits des utilisateurs (accès, rectification, effacement, portabilité)
    → Politique cookies (aucun cookie, localStorage uniquement)
  - Mentions légales (éditeur, hébergement, propriété intellectuelle)
  - Conditions générales d'utilisation (inscription, services, limitations)
  - Lien contact (email dédié : foodquant.app@gmail.com)
  - Copyright © 2026
  - Chaque lien ouvre une modale animée

6.10 - UX / Design
  - Écran de chargement animé avec messages rotatifs
  - Design mobile-first avec Tailwind CSS v4
  - Thème ambré cohérent
  - Navbar avec icônes et emojis
  - Bouton "Sauvegarder mon menu du jour" avec feedback visuel
  - Protection double-clic (disabled pendant le loading)
  - Messages d'erreur visibles pour l'utilisateur
  - Titre de l'onglet navigateur "FoodQuant"

6.11 - Backend & Performance
  - Cache fichier pour les recettes (réduit les appels API Spoonacular)
  - Cache permanent pour les traductions (réduit les appels API Claude)
  - Récupération de 10 résultats, stockage en cache, sélection aléatoire
  - Nettoyage des recettes avant sauvegarde (JSON léger ~1Ko au lieu de ~50Ko)
  - CORS configuré pour les appels cross-origin
  - Headers Authorization acceptés dans CORS

8. DIFFICULTÉS RENCONTRÉES ET SOLUTIONS
================================================================================

Problème : Écran blanc après nettoyage du localStorage
  → Solution : Ajout de fallback dans App.jsx et MaJournee.jsx

Problème : Erreur 402 API Spoonacular (limite quotidienne atteinte)
  → Solution : Verrouillage quota 1h + cache expiré + 20 recettes fallback

Problème : "could not find driver" sur Railway (pdo_mysql manquant)
  → Solution : Dockerfile avec php:8.2-cli + docker-php-ext-install pdo_mysql

Problème : Connexion MySQL Railway échouait (host interne)
  → Solution : Utilisation du host public (nozomi.proxy.rlwy.net:54599)

Problème : FrankenPHP ne renvoyait rien (page blanche)
  → Solution : Passage à php:8.2-cli avec serveur intégré PHP

Problème : Structure de dossiers dupliquée (backend-php/backend-php/)
  → Solution : Déplacement des fichiers et suppression du sous-dossier

Problème : Port mal configuré sur Railway
  → Solution : Configuration du port 8080 dans Networking

Problème : Bouton "Sauvegarder mon menu" ne fonctionnait pas
  → Cause : Erreur SQL "Data too long for column recettes_json" (JSON ~50Ko)
  → Solution : Nettoyage des recettes (~1Ko) + ALTER TABLE en MEDIUMTEXT

Problème : Recettes affichées en anglais
  → Solution : Traduction via API Claude Haiku avec cache permanent

Problème : Identifiants BDD en dur dans le code source
  → Solution : Variables d'environnement Railway + db.php sécurisé

Problème : Pas d'authentification sécurisée (userId en clair)
  → Solution : Tokens JWT signés HMAC-SHA256, durée 7 jours

Problème : Footer.jsx introuvable (erreur import)
  → Solution : Déplacement du fichier dans le dossier components/
